进入**第四阶段：指挥塔 (The Cockpit) —— 通讯协议与可视化交互设计**。

在这一阶段，我们将解决“透明度”与“可控性”的问题。由于 **TokenRun** 处理的是长耗时、高成本的任务，用户不能处于“盲等”状态。我们需要一套高效的实时通讯机制，将底层的 `Trace` 数据转化为直观的执行脉络图。

---

### 1. 通讯机制选型：双向实时流 (WebSocket + SSE)

*   **SSE (Server-Sent Events):** 用于从 Runner 向 UI 推送实时的任务进度、Token 消耗和执行日志。
*   **WebSocket:** 用于用户与 Runner 之间的双向交互，例如“点击暂停”、“修改 Prompt 后热更新”、“批准全量执行”。

---

### 2. 指挥塔核心通讯协议 (The Cockpit Protocol)

我们定义一套标准的消息格式，用于前端展示“任务流”：

#### A. 实时状态心跳 (Heartbeat Packet)
```json
{
  "type": "STATUS_UPDATE",
  "payload": {
    "task_id": "tr-2024-001",
    "phase": "SAMPLING",      // SAMPLING | WAITING_APPROVAL | FULL_RUN | COMPLETED
    "total_progress": 0.05,   // 5%
    "current_usd": 0.45,
    "estimated_total_usd": 12.50,
    "active_workers": 4
  }
}
```

#### B. 执行轨迹流 (Trace Stream)
这是最关键的 UI 逻辑：展示 **Loop Engineering** 的推演过程。
```json
{
  "type": "TRACE_EVENT",
  "payload": {
    "node_id": "translator_01",
    "input_preview": "The quick brown fox...",
    "iterations": [
      {
        "index": 1,
        "status": "REJECTED",
        "critique": "语气太僵硬，不够学术化",
        "tokens": 450
      },
      {
        "index": 2,
        "status": "ACCEPTED",
        "output_preview": "敏捷的棕色狐狸...",
        "tokens": 480
      }
    ]
  }
}
```

### 3. “1% 采样”决策台 (Approval Dashboard)

当 Runner 完成采样阶段后，会发送一个 `APPROVAL_REQUIRED` 信号。前端需要呈现一个专门的对比视图：

*   **左侧：** 任务蓝图 (Runfile) 的核心逻辑与 Prompt。
*   **中间：** 采样出的 1% 样本结果（包含 Critic 的评价）。
*   **右侧：** 经济预估（消耗量、时长预估、ROI 分析）。

**用户操作：**
1.  **Approve & Run:** 锁定指纹，开启剩下的 99%。
2.  **Edit & Resample:** 修改 Prompt，丢弃本次指纹，重新采样。
3.  **Halt:** 终止任务，保存当前的成功样本为“半成品资产”。

### 4. 核心组件设计：实时日志存证器 (Telemetry)

为了让“指挥塔”能够追溯历史任务，我们需要一个高性能的 Telemetry 组件。

```python
class TelemetryManager:
    """负责将执行日志分发到前端并持久化"""
    def __init__(self, websocket_manager):
        self.ws = websocket_manager
        self.event_log = []

    async def broadcast_step(self, task_id: str, step_data: Dict):
        """向所有已连接的 UI 推送最新的循环步骤"""
        message = {
            "task_id": task_id,
            "timestamp": time.time(),
            "data": step_data
        }
        await self.ws.send_json(message)
        self.event_log.append(message) # 也可以写入数据库/Redis

    async def emit_sample_report(self, task_id: str, report: Dict):
        """推送 1% 采样报告给用户审批"""
        await self.ws.send_json({
            "type": "APPROVAL_REQUIRED",
            "task_id": task_id,
            "report": report
        })
```

### 5. 交互式干预 (Human-in-the-Loop)

这是 TokenRun 的独特优势：**任务在奔跑时，用户可以“手动修正”。**

*   **逻辑流：**
    1.  Runner 在全量运行。
    2.  用户查看实时 Trace，发现某个子任务虽然通过了 Critic 验证，但自己不满意。
    3.  用户点击“Pause”。
    4.  用户在 UI 上直接修改该节点的 `actor_prompt_template`。
    5.  用户点击“Resume”。
    6.  **Runner 处理逻辑：** 清空未执行的待办队列，根据新 Prompt 重新生成后续任务的逻辑指纹，然后继续。
