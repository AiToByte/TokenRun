### 1. 强版本控制：提示词谱系 (Prompt Lineage)

当用户在任务运行中途修改 Prompt 时，我们不能简单地覆盖旧代码。我们需要记录每一次干预的“基因突变”。

#### A. 模型定义：Prompt 注册表
```python
class PromptVersion(BaseModel):
    version_id: str          # v1, v2, v2.1...
    hash: str                # 模板内容的哈希
    template: str            # 具体的 Prompt 内容
    parent_id: Optional[str] # 溯源 ID，记录是从哪个版本修改而来的
    change_log: str          # 用户修改的原因（人工输入或 AI 自动总结）
    stats: Dict[str, Any]    # 该版本下的平均评分、通过率、Token 效率
    timestamp: float

class TaskNode(BaseModel):
    # ... 原有字段
    prompt_registry: List[PromptVersion] # 该节点演进的所有版本
    current_version_id: str              # 当前激活的版本
```

#### B. 执行轨迹中的版本锚定
在 `ExecutionIteration`（执行迭代）中，必须强制包含 `prompt_version_id`。
*   **价值：** 这样你就可以在指挥塔上看到：*“前 50 组数据是用 v1 跑的，通过率 60%；用户干预后，后 50 组是用 v2 跑的，通过率提升到了 95%。”*

### 2. 技能固化机制 (Skill Solidification)

当整个 Runfile 执行完毕（全量 100%），TokenRun 会启动 **“炼金提纯”** 程序，将执行过程提炼为一个 **Permanent Skill (永久技能)**。

#### A. 提取“黄金路径” (Golden Path Extraction)
系统会自动分析 `Trace` 数据，剔除那些失败重试的循环，仅保留：
1.  **最终验证通过的 Prompt 版本。**
2.  **最佳性能的 Model 参数（Temperature, Top_p 等）。**
3.  **最优的 Critic 审计准则。**

#### B. 封装为 .trs (TokenRun Skill) 文件
一个固化后的技能包将包含：
*   **Manifest:** 技能描述、适用领域、预估 Token 投入产出比。
*   **Logic:** 经过版本控制筛选后的最终 `Runfile`。
*   **Tests:** 采样阶段沉淀下来的“黄金样本对”（输入-标准输出），用于未来的回归测试。

### 3. 系统集成：从“坟场”到“技能库”的闭环

我们将整个系统的生命周期整合为以下流程：

#### 1. 触发 (Ingest)
用户丢入任务 $\rightarrow$ 系统生成 `Runfile v1.0`。

#### 2. 进化 (Evolve)
在 **1% 采样** 或 **全量执行** 过程中：
*   用户发现 Critic 报错频繁。
*   用户干预，修改 Prompt $\rightarrow$ 产生 `v1.1`, `v2.0`。
*   系统实时记录不同版本的“性能分布图”。

#### 3. 固化 (Solidify)
任务结束，用户点击“Save as Skill”：
*   系统自动锁定表现最好的版本。
*   将该任务从 `Active Job`（活跃任务）转入 `Skill Vault`（技能库）。

#### 4. 复用 (Reuse)
下一次用户提交类似任务时：
*   系统检测意图，提示：*“检测到已有成熟技能‘学术文献炼金术’，是否直接套用？”*
*   此时，由于有**指纹锁定**和**版本记录**，执行将具有极高的**确定性**。
