经过对上传代码库的深度审计，并将其与我们之前讨论的**“工业级 AI 炼金厂”**设计蓝图进行对比，我为您整理了一份详尽的差异分析报告。

总体来看，项目的**核心引擎层（Core Engine）**完成度极高且质量精湛，但在**前后端闭环（UI-Backend Loop）**、**自愈机制的深度集成**以及**分布式状态同步**方面存在明显的待完成项。

---

### 一、 核心设计差异与未完成项汇总

#### 1. 人机交互闭环（Human-in-the-Loop）的断层
*   **设计要求：** 1% 采样完成后系统暂停，用户在 UI 审核通过后，Runner 继续执行剩下的 99%。
*   **代码现状：** 
    *   `core/sampling_manager.py` 实现了 `wait_for_approval`，但它在 CLI 模式下是靠 `asyncio.sleep` 轮询标志位的。
    *   `api/main.py` 中的 `approve_mission` 接口仅仅修改了内存中 `_active_missions` 字典的状态。
*   **差异点：** **缺乏信号同步机制**。API 层修改的状态无法即时通知到已经在后台运行的 `app.run_mission` 协程。目前后台任务在采样完后会陷入死循环等待，除非人工在控制台干预，Web 端的“Approve”按钮实际上是“虚晃一枪”。
*   **实现建议：** 引入 `asyncio.Event` 并在 API 层面触发该事件，或使用持久化队列（如 Redis）管理任务信号。

#### 2. “Prompt 进化实验室”功能的浅层化
*   **设计要求：** 强版本控制（Prompt Lineage），用户修改 Prompt 后，系统记录 `v1.0 -> v1.1` 并对比通过率。
*   **代码现状：** 
    *   后端有 `core/prompt_lineage.py` 逻辑，前端有版本对比 UI。
    *   `api/main.py` 提供了 `/revise` 接口，但它仅仅记录了 `prompt_revision` 字段。
*   **差异点：** **未实现“热重载采样”**。当用户点击“Edit & Resample”时，系统应该终止当前的采样任务，应用新 Prompt，并重新启动 1% 采样流。目前代码中缺乏“任务重置”的逻辑，修改后的 Prompt 无法立即应用到正在运行的任务。

#### 3. 自愈机制（Self-Healing）的离群
*   **设计要求：** 当 Critic 连续出现相似负面反馈时，元模型自动优化 Prompt 并提示用户更新。
*   **代码现状：** `core/self_healer.py` 作为一个独立的工具类存在。
*   **差异点：** **未与执行流集成**。`TROrchestrator` 虽然在 `_bounded_execute` 中调用了 `record_critique`，但并没有逻辑去触发 `generate_healing` 并在 UI 上弹出“建议修改”的消息。自愈目前处于“只收录不行动”的状态。

#### 4. 资源网关（MCP）的协议完整性
*   **设计要求：** 深度集成 Anthropic 的 MCP 协议，实现隐私脱敏和本地连接。
*   **代码现状：** 
    *   `core/mcp_server.py` 实现了一个自定义的 JSON-RPC 服务，将 Skill 暴露为工具。
    *   `gateway/` 目录下有各种 `xxx_gateway.py`。
*   **差异点：** **缺乏“MCP Client”能力**。目前 TokenRun 可以作为 MCP 服务端供 Claude 调用，但 `TROrchestrator` 无法主动连接外部的其他 MCP Server（如连接用户的 Notion Server 或 GitHub Server）。目前的 `local://` 协议是硬编码实现的，而非通过标准 MCP 协议通信。

### 二、 实现细节中的“有误”或“待优化”点

#### 1. 沙箱安全性（Sandbox Security）
*   **实现有误：** `core/sandbox.py` 和 `runner.py` 中的 `code_eval` 使用的是 `subprocess.run([sys.executable, ...])`。
*   **风险：** 这种方式虽然实现了隔离，但在工业级环境下极度危险。AI 生成的 Python 代码可以直接通过 `os` 库访问系统文件或网络。
*   **优化建议：** 应改用 `RestrictedPython` 或者在 `Docker` 容器/ `WASM` 沙箱内运行这些校验代码。

#### 2. 指纹验证（Fingerprint）的颗粒度
*   **实现偏离：** `ActorCriticLoop.verify_fingerprint` 目前只检查 `model_id` 和 `prompt_hash`。
*   **改进点：** 设计中提到过要包含 `Seed` 和 `Temperature` 的锁定。目前的实现如果用户在配置中改了 `temperature`，指纹依然能通过验证，这会导致确定性下降。

#### 3. 大规模并发下的 Token 审计延迟
*   **潜在问题：** `TokenLedger` 使用的是简单的同步加法。在高并发协程环境下，如果没有 `threading.Lock` 或 `asyncio.Lock`（在代码中未见明显针对累加器的锁），在极高吞吐量下可能存在计费竞争（Race Condition）导致账本偏差。

#### 4. 前端状态机同步（Telemetry）
*   **未实现：** 前端 `missions/page.tsx` 使用 `setInterval` 每 3 秒拉取一次。
*   **差异：** 我们设计中强调了 **WebSocket / SSE 实时的 Live Trace**。虽然后端 `api/main.py` 写了 `/ws` 接口，但它与 `TROrchestrator` 之间没有建立订阅关系。Runner 的日志吐到了 console，但没有流向 WebSocket。

### 三、 总结：目前项目与设计的符合度

| 模块 | 符合度 | 状态 | 缺失点 |
| :--- | :--- | :--- | :--- |
| **TRP 协议 (Runfile)** | 100% | 完美 | 无 |
| **Loop Engineering** | 95% | 优秀 | 自愈机制未闭环 |
| **1% Sampling** | 70% | 可用 | 前后端审批信号未对接 |
| **Privacy Redactor** | 100% | 完美 | 无 |
| **Skill Solidification** | 90% | 优秀 | 递归嵌套 Skill 的逻辑尚薄弱 |
| **The Cockpit (UI)** | 60% | 雏形 | 实时流展示和热更新逻辑缺失 |
| **Token Arbitrage** | 80% | 可用 | 动态模型路由优先级逻辑待细化 |

### 四、 后续落地建议（按紧急程度排序）

1.  **打通审批信号：** 在 `TROrchestrator` 中引入信号监听，让 Web UI 能够真正恢复被暂停的采样任务。
2.  **完善 Telemetry 链路：** 将 `TROrchestrator` 内部的 `telemetry.emit` 消息通过全局队列发送到 FastAPI 的 WebSocket 广播函数。
3.  **自愈逻辑接入 UI：** 当 `SelfHealer` 产生建议时，通过 WebSocket 推送到前端，显示在 Prompt Editor 的“建议版本”中。
4.  **强化沙箱：** 至少引入 `RestrictedPython` 或对 Subprocess 的权限做严格限制。

**总的来说，引擎部分（Engine）写得非常硬核且规范，剩下的工作主要集中在“把散落在各处的珍珠（组件）用一条丝线（信号与通讯）串起来”。**
