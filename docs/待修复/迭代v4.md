经过对优化后的代码库（v3 迭代版本）的深度审计，并结合项目最新的第一性原理——**“将 AI TOKEN 转换为可靠、高质量的输出结果”**，我为你整理了目前仍存在的**功能断层、潜在 Bug、以及待优化的工程细节**。

虽然项目已经从“实验性”走向了“工业级”，但在**闭环自动化**和**生产环境的极端稳健性**上，仍有几个关键台阶需要跨越。

---

### 一、 核心逻辑断层 (Logical Gaps)

这些是目前系统中最影响“自动化流水线”体验的缺失部分：

1.  **重放（Replay）机制的“空转”：**
    *   **现状：** `api/main.py` 实现了 `/replay` 接口，`TROrchestrator` 内部也定义了 `_replay_event`。
    *   **问题：** `Orchestrator` 的 `_bounded_execute` 循环中**没有任何地方在等待（await）或响应**这个重放信号。目前的重放请求只是在 API 内存里打了个标记，后台正在奔跑的生产线完全感知不到，不会触发“回滚”或“重卷”逻辑。
2.  **自愈（Self-Healing）的“断路”：**
    *   **现状：** `TROrchestrator` 能检测重复批评模式并发送 `HEALING_SUGGESTION` 信号给 Telemetry。
    *   **问题：** 缺乏“一键应用”的自动化逻辑。系统给出了优化建议，但用户必须手动复制 Prompt 到编辑器，再点击 Resample。作为“提炼厂”，应当实现：用户在 UI 点击“接受建议” $\rightarrow$ API 调用 `SelfHealer.apply_healing` $\rightarrow$ 自动重启采样。
3.  **多模型共识（Consensus）的性能隐患：**
    *   **现状：** `runner.py` 中实现了 `_run_consensus`，采用的是**串行**调用（`for model_name in consensus_models`）。
    *   **问题：** 如果配置了 3 个共识模型，审计时间将翻 3 倍，极大拖慢高并发吞吐。
    *   **建议：** 应改为 `asyncio.gather` 并行调用所有共识模型。

---

### 二、 潜在 Bug 与工程隐患 (Potential Bugs & Technical Debt)

1.  **资源网关的路径遍历风险：**
    *   **现状：** `file_gateway.py` 的 `save_result` 增加了 `resolve()` 校验。
    *   **隐患：** `stream_files` 方法依然直接接收 `pattern` 参数。恶意构造的 glob 模式（如 `../../**/*`）在某些操作系统下仍可能通过 `base_path` 访问到授权目录之外的文件。
    *   **建议：** 强制对所有的资源路径进行 `os.path.commonpath` 检查。
2.  **Jinja2 模板污染风险：**
    *   **现状：** `core/actor.py` 升级到了 `SandboxedEnvironment`，非常好。
    *   **隐患：** `core/drift_detector.py` 中的 `run_check` 仍然使用原生的 `Template` 类。如果用户的 `Runfile` 是从不信任的来源导入的，在运行漂移检测时可能触发任意代码执行。
3.  **Token 账本的原子性竞争：**
    *   **现状：** `TokenLedger` 使用了 `threading.Lock`。
    *   **隐患：** TokenRun 是一个高并发 `asyncio` 系统。虽然 `threading.Lock` 能保证计数器不脏，但在极高负载下，它会阻塞整个 Python 事件循环的线程。
    *   **建议：** 应改用 `asyncio.Lock`，以保持协程的非阻塞特性。

---

### 三、 与“提炼厂”定位不符的设计 (Vision Mismatches)

1.  **输出汇（Sinks）的缺失：**
    *   **现状：** 处理后的高质量结果默认通过 `FileGateway` 存为 `.refined` 文件。
    *   **距离：** 既然是“转换为高质量输出”，输出应当是可结构化的。目前的 `Runfile` 协议中没有定义“输出目的地”。
    *   **建议：** 增加 `OutputSink` 模块，支持将结果直接推送到 `DuckDB`、`Notion API` 或 `WebHook`。目前 DuckDB 只是个“零件”，没装进“流水线”。
2.  **MCP 的“伪集成”状态：**
    *   **现状：** `app.py` 支持了 `ResourceType.MCP_TOOL`，但代码里硬编码调用了名为 `list_data` 的工具。
    *   **距离：** 这违背了 MCP 的灵活性。Runfile 应该能描述具体的工具名称、参数映射（Parameter Mapping）。目前它只是为了通过测试而写的“死代码”。
3.  **ROI 报告的“静态性”：**
    *   **现状：** ROI 报告在任务结束时生成。
    *   **距离：** 作为提炼厂，用户需要在 UI 上看到**实时的价值曲线**。目前的前端 Dashboard 只是简单的数字叠加，缺乏动态的价值产出视图。

---

### 四、 待实现的进阶功能模块 (Unfinished Modules)

1.  **知识蒸馏自动化 (Automated Distillation Loop):**
    *   项目定位中提到了转换。目前系统能生成 `.trs`（逻辑资产），但还不能自动触发 `SkillSolidifier.export_fine_tune`。
    *   **理想状态：** 全量任务成功率 > 90% 后，系统自动在 `vault/` 生成一份针对该任务的训练数据集，实现“从 Token 到 模型智能”的闭环。
2.  **语义漂移的“自愈动作”:**
    *   目前 `DriftDetector` 发现漂移只能 `halt`（停止）。
    *   **理想状态：** 实现 `action: "resample"`。当检测到模型变笨或漂移时，系统自动挂起任务，生成一份新的采样报告对比，让用户决定是否更新指纹。
3.  **版本树的可视化渲染:**
    *   API 已经给了 `nodes` 和 `edges`。
    *   前端 `missions/page.tsx` 仍然只渲染了一个扁平列表。需要引入 `react-flow` 或类似的库，渲染出真正的 **Prompt 进化树**。

---

### 五、 总结：下一步优化的优先级建议

1.  **【核心闭环】** 修改 `orchestrator.py`，真正接入 `_replay_event`，让 UI 上的“重试”按钮生效。
2.  **【提炼能力】** 完善 `ResourceType.MCP_TOOL` 的动态调用逻辑，让系统能真正连接外部生态。
3.  **【安全加固】** 统一所有的 `Template` 渲染为沙箱模式，并强化 glob 路径校验。
4.  **【资产沉淀】** 将 `DuckDB` 作为 Runfile 的可选输出节点，让提炼结果直接落库。

**目前的 TokenRun 核心已经非常强壮（95% 符合设计），剩下的 5% 主要是“把写在纸上的高级协议（如 Replay, Consensus, MCP）在代码逻辑中彻底激活”。**
