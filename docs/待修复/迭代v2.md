项目定位从“Token坟场（废物利用）”转变为“**AI TOKEN 转换器（高价值提炼）**”，意味着项目的核心目标从“**消耗**”转向了“**交付**”。

在这一新定位下，项目不再仅仅是一个处理过期点数的工具，而是一个**工业级 AI 生产流水线控制器**。再次审查代码库（特别是 v1.1 迭代后的版本），我为您找出了目前实现中存在的**功能偏差、距离及未完成模块**，分为四个维度罗列如下：

---

### 一、 质量一致性与确定性（Reliability & Quality）

定位变更为“可靠、高产出”后，系统对“漂移”和“失败”的容忍度需要进一步降低。

1.  **漂移处理的“软逻辑” (Drift Handling Gap):**
    *   **现状：** `DriftDetector` 已经实现，但在 `Orchestrator` 中，检测到漂移仅会打印日志或返回一个警告字段。
    *   **偏差：** 在生产级流程中，检测到语义漂移（Embedding 相似度下降）应具备**自动熔断**或**强制重采样**的能力。目前的实现过于“软”，无法保证全量执行过程中的端到端一致性。
2.  **质量熔断机制 (Quality-based Circuit Breaker):**
    *   **缺失：** 目前只有 `BudgetExceededError`（财务熔断）。
    *   **建议：** 需要增加“质量熔断”。例如：如果连续 N 个任务的 `Critic` 评分低于 0.6，说明当前 `Runfile` 或 Prompt 在该数据集上失效，应立即停止流水线，防止无效 Token 损耗。
3.  **多模型共识验证 (Consensus Validation):**
    *   **缺失：** 对于“极高可靠”任务，单模型 Critic 可能存在偏差。
    *   **建议：** 在 `LoopConfig` 中增加 `consensus` 选项，支持调用两个不同的廉价模型（如 GPT-4o-mini + DeepSeek）进行双重审计。

---

### 二、 生产编排与自愈闭环（Orchestration & Self-Healing）

定位变更为“提炼厂”后，系统应具备更强的自动化优化能力。

1.  **自愈逻辑未“闭环” (Healing Loop Closure):**
    *   **现状：** v1.1 实现了 `SelfHealer` 并能通过 Telemetry 发送建议，但它并没有改变 `Orchestrator` 的执行流。
    *   **距离：** 理想状态是系统检测到重复错误后，弹出 UI 提示“检测到优化方案，点击应用并重启采样”。目前代码中 `apply_healing` 只是生成了新版本，并没有“热切换”运行中任务的机制。
2.  **真正的“词元套利”调度 (True Token Arbitrage):**
    *   **现状：** `CostScheduler` 主要是根据优先级选择 Batch API 或同步 API。
    *   **偏差：** 真正的提炼厂应该具备**动态路由**：如果 Actor A 连续 3 次失败，调度器应自动尝试成本接近但逻辑不同的 Actor B（如从 Claude 切换到 GPT）。目前路由逻辑是静态配置在 `ModelTier` 里的。
3.  **Skill 递归嵌套的深度实现 (Recursive Skill Implementation):**
    *   **现状：** `TaskNode` 有 `skill_ref`，`Orchestrator` 有 `resolve_skill_ref`。
    *   **实现偏差：** 目前只是把 Skill 的 Prompt 复制过来。真正的“技能积木”应该支持**子任务流嵌套**。即一个节点可以引用一个包含多个节点的 `.trs` 文件，实现“流水线套流水线”。

---

### 三、 生态连接与数据资产（Ecosystem & Assets）

定位变更为“AI TOKEN 转换器”后，输入和输出的连接能力至关重要。

1.  **MCP Client 的深度集成 (Actual MCP Usage):**
    *   **现状：** v1.1 引入了 `MCPClient` 类。
    *   **距离：** 目前 `Orchestrator` 和 `FileGateway` 依然是硬编码的 IO 逻辑，并没有真正利用 `MCPClient` 去连接外部的 MCP Server（如官方的 Google Drive 或 GitHub MCP）。
    *   **建议：** 应重构 `Resource` 模型，支持 `type: mcp_tool`，让 TokenRun 能够直接调用外部工具作为生产资料。
2.  **结构化资产持久化 (Data Asset Storage):**
    *   **偏差：** 目前处理后的结果通过 `FileGateway` 存为 `.refined` 文件。
    *   **距离：** 作为一个提炼厂，输出应支持存入结构化数据库（如 DuckDB 或本地 Postgres），以便后续分析。代码中虽然有 `duckdb_gateway.py`，但在 `Orchestrator` 的主流程中并未深度集成。
3.  **知识蒸馏的自动化 (Automated Distillation):**
    *   **现状：** `SkillSolidifier` 提供了 `export_fine_tune` 方法。
    *   **距离：** 这是一个主动调用的函数，未集成到 API 或 UI 中。应实现在 UI 上一键“导出本次任务为训练集”。

---

### 四、 交互与运维（Cockpit UI/UX）

定位变更为“工业级框架”后，运维监控的专业感需要加强。

1.  **WebSocket 广播的稳定性 (WS Reliability):**
    *   **现状：** `api/main.py` 的 WebSocket 只是简单的内存 List。
    *   **问题：** 缺乏订阅机制（Pub/Sub）。当多个用户或多个页面打开时，消息分发可能会混乱或丢失。在高负载任务下，需要一个更稳健的消息队列来处理 Telemetry 数据。
2.  **可视化版本树 (Visual Version Tree):**
    *   **现状：** 前端只是列出了 Prompt 的版本历史。
    *   **距离：** 既然有了 `PromptLineage`，用户需要看到一个**树状图**，清晰地看到从 `v1.0` 演进到 `v1.2` 的路径，以及每个分支的质量指标。
3.  **时间旅行调试器的完整性 (Time-travel Debugger Gap):**
    *   **现状：** 前端实现了滑动条。
    *   **偏差：** 目前只能查看历史的轨迹数据，无法做到“修改历史并重新执行（Replay from here）”。这在工业调试中非常重要。

---

### 总结：后续开发的优先级建议

1.  **第一优先级（闭环优化）：** 实现 `SelfHealer` 与 UI 的交互闭环，让 Prompt 优化真正能一键生效。
2.  **第二优先级（质量护栏）：** 在 `Orchestrator` 中加入“质量熔断”逻辑，强化 `DriftDetector` 的干预权限。
3.  **第三优先级（协议升级）：** 将 `gateway/` 下的本地 IO 逻辑全面转向由 `MCPClient` 驱动，真正融入 MCP 生态。
4.  **第四优先级（资产化）：** 打通 `DuckDB` 的自动存入，让 Token 消耗的结果直接变成结构化的本地数据库资产。

目前的 **TokenRun** 已经是一个非常扎实的**内核**，后续的迭代应集中在**“增强控制力”**和**“拓宽连接力”**上。
