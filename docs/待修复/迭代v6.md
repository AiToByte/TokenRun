### 二、 存在的偏离度与待优化的“最后一公里”

虽然完成度极高，但从“极致工业化”角度看，仍有三个细微的偏离或优化空间：

1.  **隐私还原映射的非持久化隐患：**
    *   **现状：** `PrivacyRedactor` 的映射表（Vault）仅存在于内存。
    *   **偏离：** 如果一个处理 100 万 Token 的长任务在 90% 时因系统 OOM 崩溃，重启后虽然有 `Persistence` 记录，但映射表已丢失，导致之前生成的占位符无法在本地还原。
    *   **建议：** 增加一个 `Resource` 级的 `PersistentRedactor`，将 Vault 定期快照到本地 `tokenrun_traces.db` 的另一个表中。

2.  **Telemetry 分层推送的带宽优化：**
    *   **现状：** 已经定义了 L1-L3 等级。
    *   **优化：** 在 `api/main.py` 的 WebSocket 广播中，目前尚未根据等级过滤消息。当全量任务高通量运行时，大量的 L3 轨迹数据可能会撑爆前端 WS 连接。
    *   **建议：** 在 WS `subscribe` 动作中允许客户端指定 `level`，后端按需推送。

3.  **MCP 认证的缺失：**
    *   **现状：** `mcp_client.py` 已经支持工具调用。
    *   **偏离：** 目前的连接是明文的 `http://`。在实际连接企业级 MCP Server 时，通常需要 Header 认证（如 API Key）。目前协议中暂未定义认证载荷。
